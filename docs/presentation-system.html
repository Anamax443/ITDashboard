Why the system exists

From manual checks to an automatic overview

🔒 This version is anonymized. All names, IP addresses, device names and e-mails are replaced with generic examples (e.g. Site A, 10.0.0.x, PC-001).

Running a larger IT estate (tens to hundreds of computers, several sites, network gear, printers) without a tool relies on manual checks and scattered scripts. That means:

  • problems are found late — once a user reports them,
  • there is no single inventory of what is actually on the network,
  • repetitive manual work (inventories, disk / service / printer checks).

Goal

One dashboard that collects data from many sources on its own, keeps it in a database, visualizes the state and proactively warns about a problem before anyone reports it.

What the system covers

Every agenda in one place

🖥 Computers & servers

Availability, OS, packet loss & latency, owner and notes.

🌐 Network inventory

Devices on the network merged by MAC address from several sources.

💾 Disks

Free space, critical thresholds, e-mail alert for key machines.

⚙ Services

Monitoring of critical and ordinary system services.

🔌 Ports

Reachability of key TCP ports from outside (network → firewall → service).

🖨 Printers

Online/offline state, supply levels, alert on outage.

Each agenda has its own collector, a dashboard visualization and optional e-mail alerts with shared settings (thresholds, recipients, maintenance window, throttling).

Collection

An application server periodically reads from network devices and the directory service, pings hosts and reads SNMP from printers. Fully configurable, no secrets in code.

Storage

A structured SQL database with history. The inventory is keyed by MAC address (a permanent identity) — the IP is only a temporary "where it lives".

Presentation

A web dashboard with filtering and management summaries, plus automatic e-mail reports and alerts via the corporate SMTP relay.

Key capability

Network inventory: merging many sources by MAC

No single source sees everything. The system combines several views and merges them by MAC address into one record per device:

SourceWhat it adds
DHCP recordsdynamic and reserved devices + names
ARP tablestatically addressed devices outside DHCP
Active scannames (NETBIOS/DNS) and not-yet-seen devices

Duplicates are impossible by design: one row per (site, MAC), merge-and-enrich, never a new duplicate.

Robust collection

Data is read both live over an API and from file snapshots the network device writes for itself on a schedule. The snapshot doubles as a freshness signal — when it stops "breathing", the system knows something is wrong and raises an alert.

Identity = MAC

A device keeps its category, name and IP history even when its IP changes. "MAC = the permanent ID, IP = the temporary address."

Proactivity

E-mail alerts, not problem hunting

Instead of walking the floor, the system sends a structured e-mail the moment a defined state occurs. Every alert shares a sensible model:

  • Debounce — won't fire on a momentary blip,
  • Throttling — reminds at a sensible interval, never floods,
  • Maintenance window — stays quiet during planned work,
  • Per-site / per-machine — exceptions, so what is still being rolled out doesn't "scream".

Under the hood

Technology, security, operations

Technology

Node.js / TypeScriptREST API SQL databaseWeb frontend SNMPDirectory service SMTP

Operations

  • runs as a service on an internal server,
  • automatic deployment after an approved change (CI: tests → build → migrations),
  • a health-check endpoint for external monitoring.

Security

  • Read-only service accounts talk to network devices for reading only,
  • access restricted to allow-listed addresses,
  • passwords and secrets are encrypted, never in code or the repository,
  • data stays inside the corporate network.

Least-privilege by design: even on a leak, the service account can only read, and only from defined locations.
