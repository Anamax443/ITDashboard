Funkční rozsah

  • Eventlog napříč flotilou — Warning/Error/Critical z každého PC do centrální DB; filtrování, vyhledávání, časové grafy, žebříček „noisy" PC.
  • AD-synchronizovaný inventář — OS, poslední přihlášení, OU cesta, operátorem řízený monitor toggle (perzistuje napříč syncs). Monitor a Exclude se vzájemně vylučují — zapnutí jednoho zruší druhý (vynuceno na serveru u per-řádek i hromadných akcí).
  • Monitoring disků — periodický DCOM sken, konfigurovatelné prahy (% / GB), per-tier scope písmen disků.
  • Email alerty disků — per-PC opt-in (sloupec 📧 Disk v Počítačích, volitelně konkrétní písmena C,F); po skenu pošle report o kritických discích, throttle 1× za nastavený interval. SMTP relay, příjemci a četnost v Settings; defaultně vypnuto.
  • Email alerty služeb — per-PC opt-in (sloupec 🔔 Služby) na konfigurovatelný seznam kritických služeb (NTDS, DNS, Kdc, … + wildcardy) s whitelistem; email když je kritická služba mimo Running, s ochranou proti flappingu (časový debounce + okno údržby). Sdílí SMTP/příjemce s disky; defaultně vypnuto.
  • Services collector + policy — Auto + neběžící služby, klasifikace drift vs. policy tabulka, ExitCode capture, GPO PS export.
  • Performance events — pomalý boot/shutdown/standby/resume z Diagnostics-Performance kanálu s pojmenovanými culprity.
  • Klasifikace dostupnosti — každý běh: online / offline / rpc_unavailable / access_denied.
  • Activity log — terminálový live view (filtr, pauza, copy) + perzistentní historie.
  • Porty — záložka s živou mřížkou dostupnosti nakonfigurovaných TCP portů per stroj (otevřeno/zavřeno + latence); sonda na pozadí, ruční obnova a per-řádek živý ping (konzolové okno).
  • Zařízení (Devices) — víceztrojový inventář slučovaný podle MAC (sloupec source = dhcp/arp/scan/unifi/share): MikroTik DHCP (dynamické leasy + statické rezervace), ARP tabulka routerů, aktivní sken podsítí z aplikačního serveru, UniFi kontrolér a sdílené/USB tiskárny. Identita na každý router tahá tři REST zdroje: GET /rest/ip/dhcp-server/lease, GET /rest/ip/arp a nově POST /rest/tool/ip-scan (aktivní sken běžící na routeru → NETBIOS + reverzní DNS + SNMP jména zevnitř podsítě). Konfigurovatelné rozsahy skenu (CIDR/wildcard, volitelně Site=, ! vyloučí podsíť); discovery cache; vzdálené podsítě přes router ARP. Párováno s AD počítači podle hostname/IP; per-zařízení kategorie s auto-návrhem a operátorem editovatelný název (dle MAC).
  • Rozlišení názvů + auto-návrh tiskáren — NetBIOS (nbtstat) rozlišení názvu zařízení → auto-návrh tiskáren podle prefixů (NPI/BRN/BRW/RNP/KMBT); operátorem editovatelný název zařízení dle MAC. Sloupec „Type" (statický/dynamický), AD-odvozený předvýběr pc/server, generická kategorie „Printer", operátorem konfigurovatelné kategorie.
  • Packet loss + latence per zařízení — kompaktní sloupec ms / % (např. <5/0) se ztrátovostí a průměrnou latencí; laditelné prahy „problému"; dlaždice „Loss / latence" na homepage + filtr „jen problémy".
  • Email alerty tiskáren — agenda upozornění na offline tiskárny; dlaždice „Tiskárny" na homepage; proklik z IP tiskárny na webové UI se serverovým proxy obcházejícím certifikát (nově zapnuto defaultně).
  • Stav tiskáren (náplně) — per-tiskárna hladiny inkoustu / toneru / maintenance boxu / drumu / beltu čtené přímo z tiskáren přes SNMP Printer-MIB (primární cesta, jednotná napříč HP/Epson/Brother) s cíleným HTTP fallbackem pro číselné % toneru Brother a maintenance box Epsonu. Nová záložka „Stav tiskáren", homepage dlaždice „🖨 Náplně" a vlajka náplní v záložce Zařízení. Vlastní SNMP v1 klient (node:dgram, bez závislosti); sonduje jen zařízení kategorie tiskárna.
  • Záložka „Databáze" — velikost DB + footprint per tabulka.
  • Per-PC „Obnovit teď" — jednorázová obnova jednoho stroje napříč všemi checky: disky, služby, eventlog, perf, porty.
  • Sběr a analýza pádů (BSOD) — server tahá kernel minidumpy z monitorovaných PC (\\PC\C$\Windows\Minidump, dedup, nikdy MEMORY.DMP) do DB a samostatný worker je analyzuje přes cdb (Debugging Tools) — STOP kód, název bugchecku, horká funkce a vinící proces/modul. Záložka „Pády" s agregacemi (STOP rozpad / top viník / opakované) a status pruhem (poslední sběr / poslední zápis do SQL / příští plánovaný běh, GET /crashes/status), detail s celým výstupem cdb a stažením .dmp, a tisknutelný report „Analýza pádu" — světlý defaultně s přepínačem světlý/tmavý a UTF-8 (bez mojibake), s vysvětlením odvozeným z dumpu (co se stalo / technicky / doporučení), které rozlišuje chybu jádra Windows od ovladače třetí strany a u pádů z poškození paměti ukazuje na vadný ovladač / špatnou RAM. Klientský C$ přes lokální-admin service accountu (Server Admins), bez eskalace. Defaultně vypnuto.
  • Přizpůsobitelný dashboard — operátorský režim úprav (✏️) rozmístí každou dlaždici podle písmene sloupce + čísla řádku (např. B3) ve dvou velikostech (plná / poloviční), umí dvě poloviční na sobě v jedné buňce (horní/dolní slot) a nadpis ke každému řádku; rozložení i nadpisy se ukládají do nastavení a po refreshi se obnoví. Dlaždice „PC v problémech" a „Operační systémy" jsou plnohodnotné dlaždice mřížky.
  • Retenční politika s ručním spuštěním — přehledová tabulka v Settings; každý řádek má tlačítko ▶ (plus Označit / Odznačit / Spustit označené). I tři device tabulky (duchové leasů, IP historie, ping vzorky), jinak mazané jen v MikroTik cyklu, jdou spustit zvlášť.
  • Zdraví komunikace (Komunikace) — homepage dlaždice 📶 s jedním celkovým stavem (ok / degraded / down) agregujícím SQL, MikroTik REST API, FTP stahování, EventLog kolektor, e-mail (SMTP/M365) a UniFi; rozbalovací panel s řádkem na kanál (stav, detail, poslední chyba, poslední OK). Endpoint GET /system/comms.
  • WAN monitor (pobočky + internet) — vždy viditelný pruh „Komunikace na pobočky a internet": server živě pinguje router každé pobočky + internetový cíl na latenci a packet loss, jeden barevně kódovaný chip na lokalitu (GET /system/wan). Volitelný (defaultně vypnutý) centrální test rychlosti stahování z internetu přes paralelní streamy měřený ze serveru (↓ X Mb/s na chipu Internet). Per-pobočkový speed test přes router není použitelný (CPU-bound), pobočky tak ukazují jen latenci + ztrátovost.
  • Měření linky (⚡) — server přes SMB zapíše a přečte soubor na C$ každého PC a spočítá rychlost nahoru/dolů (Mb/s) + latenci = reálná kvalita drátěného spoje (odhalí vadné kabely / 100Mb porty v gigabitové síti). Cíluje IP / rozsah / vše (aktivní PC/NTB/servery). Řaditelná/filtrovatelná mřížka historie (operátor / rozsah / datum), živá konzole, export CSV/HTML/PDF, proklik IP → Počítače, sloupce Hostname / Latence / Cykly. Plánovač v Nastavení (interval, hodinové okno, výjimky, cykly, název souboru); homepage dlaždice ⚡ Linky. Endpointy /system/linkspeed/*.
  • Settings — frekvence, časové okno, zapnuté checky, prahy — aplikováno živě bez restartu služby.

Žádné duplicity (slučování podle MAC + primární klíč + COALESCE upsert). Soubor je primární zdroj leasů; REST zůstává jako doplněk pro NETBIOS jména z ip-scanu (ta do souboru dumpnout nejdou). Lokality se souborem: mikrotik.ftp_sites (zatím jen Brno).

  • Sběr identity zařízení tahá na každý MikroTik tři REST zdroje (dříve dva): GET /rest/ip/dhcp-server/lease (DHCP leasy), GET /rest/ip/arp (IP+MAC vč. statických mimo DHCP) a nově POST /rest/tool/ip-scan — aktivní sken běžící na routeru přes jeho vlastní /24, vracející NETBIOS + reverzní DNS + SNMP jména. Email alerty disků

  • Per-PC opt-in v Počítačích (sloupec 📧 Disk) + volitelný výběr konkrétních písmen disků (C, C,F); prázdné = všechny in-scope disky.

  • Odesílání: přes Microsoft 365 Direct Send (axima-cz.mail.protection.outlook.com:25, STARTTLS, bez hesla, From @axima.cz) — ověřeno do Inboxu. Porty: 25 = Direct Send (bez auth, jen vlastní doména), 587 = SMTP AUTH (login), 465 = SMTPS (legacy).
  • Backend: services/alerts.ts (nodemailer) hooknutý do disk collectoru; migrace 028 + 029. Vše v Settings/DB, nic v .env.

Email alerty služeb
