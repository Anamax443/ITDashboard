ITDashboard — Documentation

Internal IT operations dashboard for the AXINETWORK domain. Provides cross-fleet eventlog visibility, AD-synced computer inventory, disk space monitoring, and per-PC reachability classification for ~225 domain machines.

Overview

ITDashboard centralises four IT-operator workflows in one dark-themed dashboard:

  • Cross-fleet eventlog visibility — pulls Warning/Error/Critical events from every monitored PC into a central DB; dashboard provides filtering, search, sorting, time-bucket charts, and noisy-PC ranking.
  • AD-synced inventory — keeps an up-to-date list of domain computers with OS, last logon, OU path, and operator-controlled monitor toggle (persists across syncs).
  • Disk space monitoring — periodic DCOM-based scan of all monitored PCs; configurable thresholds (% free or GB free or either); colored progress bars per drive.
  • Per-PC reachability classification — every collector run categorises each PC as online / offline / RPC fail / access denied, surfaced as clickable dashboard cards that drill down into filtered lists.

Single Node.js API service on the internal admin server + per-PC Electron client (or browser via Vite dev) for IT specialists.

Deployment workflow

Push-to-deploy via GitHub Actions self-hosted runner. No manual server intervention.

local edit (d:/git/ITDashboard) ↓ git push origin main GitHub Actions ↓ self-hosted runner on 10.8.2.213 picks up job robocopy → C:\Apps\ITDashboard (runtime location) ↓ npm install + typecheck + build + migrate ↓ sc stop ITDashboardAPI → poll for STOPPED state → sc start service restarts on new dist/, in-memory state cleared
Commit hash in topbar — the deployed commit SHA is shown in the dashboard topbar (clickable → GitHub commit page). If after a push the hash doesn't change, the deploy didn't fully apply — check Actions logs.
Service control ACL gotcha — the runner is svc-itdashboard, which by default cannot stop/start a Windows Service it itself hosts. We grant explicit Stop/Start rights via sc sdset as a one-time setup step. Without it, deploys succeed at robocopy/build but the service keeps running the old code.

Components

Server (apps/server)

  • Fastify + TypeScript HTTP API on port 4000
  • MSSQL via msnodesqlv8 driver (Windows-native, SSPI Integrated Auth; the pure-JS tedious driver fails with "untrusted domain" in domain environments)
  • Background tasks: unified periodic checks scheduler (eventlog / disk / services / perf), retention purge
  • PowerShell child processes for AD queries (Get-ADComputer), eventlog reads (Get-WinEvent -ComputerName), disk reads (Get-CimInstance via DCOM session)
  • Node-side TCP probe to port 135 before each PS spawn — fail-fast for offline targets, clean classification
  • In-memory ring buffer (500 entries) for activity log, polled by dashboard

Desktop client (apps/desktop)

  • Electron + React + Vite + TypeScript
  • Pure consumer of the API — no business logic, no direct AD/DB access
  • Auto-refresh dashboard every 30s; collector status polled every 2s during in-flight runs; activity log polled every 2s
  • 7 navigation tabs: Dashboard, Events, Computers, Services, Perf, Activity, Settings (+ external Docs link)

Perf tab

Surfaces slow boot / shutdown / standby / resume records pulled from the Microsoft-Windows-Diagnostics-Performance/Operational channel. This is not a continuous CPU graph — only the outliers Windows itself flagged as degraded, with the named culprit (process / service / driver / device) and how long it took.

  • 5 summary cards: affected PCs, boot events, shutdown events, standby events, resume events (selectable time range: 24h / 7d / 30d / 90d)
  • Top culprits — aggregated by culprit name across the fleet, with event count, distinct PC count, average and max time
  • Most affected PCs — by total event count, with boot/shutdown breakdown, average boot time, last event timestamp; clicking a row filters the events table to that PC
  • Recent events — full table sorted newest first: When, Computer, Category, Event ID, Total time, Degradation, Culprit (friendly name preferred, raw name in tooltip)
  • ▶ Run perf scan button — same one-shot sweep the scheduler runs

Event ID ranges: 100–199 boot · 200–299 shutdown · 300–399 standby · 400–499 resume. Default channel retention on each PC is a small ring buffer (~1 MB), so we sweep into SQL to preserve history. Cold-start sweep pulls last 7 days; incremental sweeps pull only what's new since the last collection per PC.

Activity tab

Terminal-style live log, polled every 2s:

Settings tab

Eventlog collector

Background job that pulls Warning/Error/Critical events from every monitored PC.

How it reaches each PC

  1. Node-side TCP probe to ${name}:135 with 2s timeout. If fails → OFFLINE status, skip PS spawn entirely.
  2. Spawn powershell.exe -NoProfile -NonInteractive -Command "$ErrorActionPreference='Stop'; $OutputEncoding=[UTF8]; Get-WinEvent -ComputerName '${name}' -FilterHashtable @{ LogName='System','Application'; Level=1,2,3; StartTime=$startTime } | ConvertTo-Json"
  3. PS uses MS-EVEN6 RPC over SMB (enabled by default on domain-joined Windows; doesn't require WinRM listener)
  4. Failures classified by exception message:
    • RPC server is unavailable → rpc_unavailable (TCP/135 open but RPC layer blocked, typically firewall)
    • Access is denied → access_denied (RPC OK but Event Log Readers membership missing)
    • No events found → online with 0 events (counted as success)
    • otherwise → unknown

Transport

Uses New-CimSession -SessionOption (-Protocol Dcom) explicitly. The default Get-CimInstance transport is WinRM (port 5985), which is rarely enabled fleet-wide. DCOM uses the same RPC stack as Get-WinEvent → consistent reachability semantics across both collectors.

Thresholds

Configurable in Settings. Three modes:

  • Percent free only — flag if free space % below threshold
  • GB free only — flag if free bytes below threshold
  • Either (most strict) — flag if either threshold tripped (recommended for mixed fleet — small workstations want %, large servers want GB)

Services collector + policy

Detects Windows services with StartMode = Automatic but State ≠ Running on every monitored PC. Filters legitimate cases (Trigger Start, Delayed Start, per-user instances) and compares the rest against a policy table.

SettingDefault
ScheduleSelected in Settings → Periodic checks (default included)
TransportWin32_Service via DCOM session (same RPC as eventlog/disk)
Trigger-start detectionRegistry HKLM\SYSTEM\CurrentControlSet\Services\<svc>\TriggerInfo subkey exists
Delayed-start detectionRegistry DelayedAutostart = 1
Per-user detectionService name matches _[a-f0-9]{4,12}$ (LUID suffix)

Service policy + drift

service_policy table holds rules: pattern (glob with *), expected_start_mode, expected_state, priority (lower wins), reason. Seeded with sensible defaults (Google updaters → Manual, CCAgent → Auto+Running, TrustedInstaller → Manual, etc).

After every scan, each problem is classified:

  • OK — service matches a policy and complies with it
  • Drift — service matches a policy but doesn't comply (e.g. CCAgent should be Running, is Stopped)
  • Unclassified — no policy rule matches; operator review required

UI: Services tab

  • By PC view (default) — flat list of all problems with computer / service / state / start type / drift
  • By service view — aggregated; shows # PCs / Drift / OK / Unclassified per service name — useful for "this service is stopped on 87 PCs"
  • Toggle 📊 By service / 📋 By PC
  • Filters: Hide trigger-start (default ON), Hide delayed-start, Hide per-user (default ON), Hide compliant
  • 🔧 Scan services button with live Scanning… Ns status + button text

GPO export

📤 GPO script button generates a PowerShell file (itdashboard-service-policy.ps1) that applies your current policy. Intended for:

  • GPO Computer Startup Script — Computer Configuration → Policies → Windows Settings → Scripts → Startup → PowerShell tab → Add
  • Ad-hoc run — PsExec / RDP to a single PC as admin, run the script

Script iterates each policy pattern and calls Set-Service -StartupType for every matching local service. Idempotent and safe to re-run.

EventData parsing

The PowerShell collector calls $e.ToXml() on every event and parses the EventData/Data elements by Name attribute, extracting:

  • TotalTime — total operation time in ms (stored as total_time_ms)
  • DegradationTime — extra time vs baseline in ms (degradation_ms)
  • Name — raw culprit identifier (culprit_name: process exe, service name, driver name)
  • FriendlyName — display name when present (culprit_friendly)

Server SKU gotcha

The Diagnostics-Performance channel is on by default on Windows 10/11 client, but off by default on Windows Server. Get-WinEvent on a disabled channel returns "There is not an event log on the X computer that matches". The collector matches this error pattern, classifies it as channel-disabled (separate counter, not a failure), and skips silently — no per-PC noise in the activity log, one aggregate line at end of run (e.g. "Perf scan done: 142 OK / 3 fail / 24 channel-disabled / +18 events").

To enable across the server fleet, push a GPO computer-startup script that runs:

wevtutil sl Microsoft-Windows-Diagnostics-Performance/Operational /e:true

Same pattern as the Services GPO script export. The channel starts recording on the next boot/shutdown/standby/resume cycle.

Why not continuous CPU history?

Windows does not natively retain a continuous CPU usage curve. Performance Monitor Data Collector Sets are off by default; SRUM stores per-process daily aggregates but isn't WMI-accessible. The Diagnostics-Performance channel was chosen because it is on by default, gives named-culprit attribution per incident, and reuses the existing RPC path with zero agent install. The tradeoff: you see Windows' opinion of which events were "slow" — the threshold is not configurable — and you don't see sustained workday usage, only diagnosed degradations.

This is the observer-not-executor pattern applied to telemetry: ITDashboard never installs an agent, never enables polling on the target, and never holds the target's truth — it only reads what Windows itself decided to record, on its own schedule.

AD sync

Calls Get-ADComputer -Filter * -Properties OperatingSystem, LastLogonDate, DistinguishedName against the domain controller, MERGE-upserts into computers table.

  • Computers no longer present in AD are soft-disabled (enabled = 0) instead of deleted → preserves historical events.
  • AD sync intentionally does not touch monitor_enabled — operator decisions persist across syncs.
  • OU path parsed from DistinguishedName: CN=PC1,OU=Sales,OU=Notebooks,DC=axinetwork,DC=loc → axinetwork.loc/Notebooks/Sales
  • History of every sync stored in ad_sync_runs (fetched/inserted/updated/removed counts + duration + error if any)
  • Currently triggered manually from Computers tab → ↻ Sync from AD. Daily auto-sync planned.

Troubleshooting

Topbar SHA doesn't match latest pushed commit

Cause: deploy ran but service didn't restart cleanly (race in net stop / net start) OR service account lacks Stop/Start ACL on its own service. Fix: deploy.yml uses sc stop + polling for STOPPED state since commit 4779579; the explicit ACL grant is documented in SETUP-SERVER.md.

Collector says "Connecting to remote server failed"

Cause: old code used WinRM/PSRemoting. Fix: already deployed — uses Get-WinEvent -ComputerName (RPC). If still failing fleet-wide, see permissions section.

Disk scan says "WinRM cannot connect" or "Get-CimInstance: cannot connect to destination"

Cause: old Get-CimInstance used default WinRM transport (port 5985). Fix: already deployed — uses explicit DCOM session option, same RPC stack as eventlog collector.

Czech diacritics garbled in event messages

Cause: PowerShell stdout defaulted to Windows-1250, Node read as UTF-8. Fix: already deployed — PS scripts set $OutputEncoding = [Console]::OutputEncoding = [UTF8] at start.

Some PCs return "Access denied" — not all

That's the classification working — it means: TCP/135 OK, RPC OK, but ACL missing on that specific PC. Either:

  • That PC is in an OU not covered by your Restricted Groups GPO
  • The PC was provisioned before the GPO existed and needs gpupdate /force
  • Local group membership was manually overridden

"A security package specific error occurred"

Kerberos / SSPI authentication issue specific to that target. Common causes: duplicate SPN registered, stale computer account, time skew > 5 min, two A-records pointing to the same host. Try: Test-ComputerSecureChannel -ComputerName X -Verbose; reset the computer account password if it returns false.

SQL connection fails with "untrusted domain"

Cause: using pure-JS tedious driver instead of msnodesqlv8. Fix: already in code — pool.ts imports mssql/msnodesqlv8.js.

GitHub Actions workflow fails immediately with UnauthorizedAccess on .ps1

Cause: AXINETWORK Group Policy enforces AllSigned ExecutionPolicy. Runner saves each step as temp .ps1, which can't load. Fix: already deployed — deploy.yml uses defaults.run.shell: cmd.

Operator-set Monitor checkbox keeps getting reset

It shouldn't — AD sync explicitly does not touch monitor_enabled. If a PC's checkbox reverts, check if someone else is editing via direct DB or if a custom script is running. The PATCH endpoint always stores the new value verbatim.
