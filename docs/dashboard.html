ITDashboard — DokumentaceITDashboard — Documentation

Interní IT provozní dashboard pro doménu AXINETWORK. Poskytuje napříč flotilou viditelnost eventlogu, AD-synchronizovaný inventář počítačů, monitorování diskového prostoru a per-PC klasifikaci dostupnosti pro ~225 doménových strojů. Internal IT operations dashboard for the AXINETWORK domain. Provides cross-fleet eventlog visibility, AD-synced computer inventory, disk space monitoring, and per-PC reachability classification for ~225 domain machines.

PřehledOverview

ITDashboard centralizuje čtyři pracovní postupy IT operátora v jednom dashboardu (tmavý/světlý motiv):

  • Eventlog napříč flotilou — stahuje události úrovně Warning/Error/Critical z každého monitorovaného PC do centrální DB; dashboard nabízí filtrování, vyhledávání, řazení, časové grafy a žebříček "noisy" PC.
  • AD-synchronizovaný inventář — drží aktuální seznam doménových počítačů s OS, posledním přihlášením, OU cestou a operátorem řízeným monitor toggle (perzistuje napříč syncs).
  • Monitorování disků — periodický DCOM skenování všech monitorovaných PC; konfigurovatelné thresholdy (% volného nebo GB volného nebo obojí); barevné progress bary per disk.
  • Klasifikace dostupnosti per PC — každý běh collectoru kategorizuje každý PC jako online / offline / RPC fail / access denied, prezentováno jako klikatelné dashboard karty pro drill-down.

ITDashboard centralises four IT-operator workflows in one dark-themed dashboard:

  • Cross-fleet eventlog visibility — pulls Warning/Error/Critical events from every monitored PC into a central DB; dashboard provides filtering, search, sorting, time-bucket charts, and noisy-PC ranking.
  • AD-synced inventory — keeps an up-to-date list of domain computers with OS, last logon, OU path, and operator-controlled monitor toggle (persists across syncs).
  • Disk space monitoring — periodic DCOM-based scan of all monitored PCs; configurable thresholds (% free or GB free or either); colored progress bars per drive.
  • Per-PC reachability classification — every collector run categorises each PC as online / offline / RPC fail / access denied, surfaced as clickable dashboard cards that drill down into filtered lists.

Jediná Node.js API služba na interním admin serveru + per-PC Electron klient (nebo browser přes Vite dev) pro IT specialisty. Single Node.js API service on the internal admin server + per-PC Electron client (or browser via Vite dev) for IT specialists.

Konfigurace a přenositelnostConfiguration & portability

Configuration is fully externalized into apps/server/.env. The committed template apps/server/.env.example is the single source of truth for which keys exist — the source tree itself carries no IP addresses, hostnames, domain or service-account names. To deploy somewhere else you change .env only (plus a few GitHub Actions repository Variables for the deploy job: SQL_HOST, SQL_INSTANCE, SQL_DATABASE) — never the code.

Required server keys:

  • SQL_HOST — SQL Server host
  • SQL_INSTANCE — named instance (empty for default instance)
  • SQL_DATABASE — database name
  • AD_LDAP_URL — LDAP endpoint for auth binds
  • AD_LDAP_DOMAIN — NETBIOS / DNS domain used for binds (no hardcoded default — must be set)
  • AD_LDAP_BASE_DN — base DN for group / membership lookups
  • AD_EDIT_GROUP — AD group required for edit-tier actions

Client API base resolution:

  • Browser — the UI is served by the API and uses relative URLs; it needs no API base URL and no hardcoded host.
  • Electron client — the packaged desktop app sets VITE_API_BASE at build time to point at the API host.
  • Protocol-handler installer — the API base is auto-injected by the server at download time (rewritten to the host the browser fetched the installer from), so it is never hardcoded in the source.

Konfigurace je plně externalizovaná do apps/server/.env. Verzovaná šablona apps/server/.env.example je jediný zdroj pravdy o tom, jaké klíče existují — samotný zdrojový strom neobsahuje žádné IP adresy, hostnames, doménu ani jména service accountů. Pro nasazení jinam měníš jen .env (plus pár GitHub Actions repository Variables pro deploy job: SQL_HOST, SQL_INSTANCE, SQL_DATABASE) — nikdy ne kód.

Povinné server klíče:

  • SQL_HOST — host SQL Serveru
  • SQL_INSTANCE — pojmenovaná instance (prázdné pro default instance)
  • SQL_DATABASE — název databáze
  • AD_LDAP_URL — LDAP endpoint pro auth bindy
  • AD_LDAP_DOMAIN — NETBIOS / DNS doména pro bindy (žádný hardcoded default — musí být nastaveno)
  • AD_LDAP_BASE_DN — base DN pro lookup skupin / členství
  • AD_EDIT_GROUP — AD skupina vyžadovaná pro edit-tier akce

Resolution API base na klientu:

  • Browser — UI je servírované přímo API a používá relativní URL; nepotřebuje žádnou API base URL ani hardcoded host.
  • Electron klient — zabalená desktop app nastavuje VITE_API_BASE v build čase tak, aby ukazovala na API host.
  • Installer protocol handlerů — API base injektuje server při stažení (přepíše ji na host, ze kterého si browser installer stáhl), takže není nikdy hardcoded ve zdroji.

PC v problémechProblem PCs

A new second row of tiles on the homepage flags PCs in trouble based on accumulated events. Each PC gets a "damped blend" score over a window (default 14 days, configurable):

  • For each event type (provider + event ID) only up to a capped number of occurrences counts (default 20) — so one chatty driver can't sink a PC on its own.
  • Occurrences are weighted by severity: critical ×10, error ×3, warning ×1.
  • Plus a bonus for how many different errors and how many days had errors.
  • A PC scoring at/above the "Risk" threshold (default 600) is counted in the red "Problem PCs" tile. (The threshold is tunable and was calibrated against the live fleet; a lower "Watch" tier exists in the settings but is not shown as a tile.)
  • Click a tile → an inline breakdown table expands below it (it is hidden by default): score, critical/error/warning counts, number of error types and number of days with errors. Click the tile again or ✕ to collapse.
  • In the table the numbers are click-throughs to the Events tab: the score opens all error events for that PC over the window (and its tooltip explains how the score is computed); crit / err / warn open events for that PC filtered to that severity. The PC name jumps to the Computers tab.
  • Everything — window, cap, thresholds — is tunable in Settings → "Problem PCs".

Nová druhá řada dlaždic na úvodní stránce vytipuje počítače v problémech podle kumulace eventů. Každé PC dostane skóre („tlumený mix") za okno (výchozí 14 dní, konfigurovatelné):

  • Za každý typ eventu (provider + ID) se započítá nejvýš stropový počet výskytů (výchozí 20) — takže jeden ukecaný ovladač PC sám o sobě nepotopí.
  • Výskyty jsou váženy závažností: critical ×10, error ×3, warning ×1.
  • Plus bonus za počet různých chyb a počet dní s chybami.
  • PC se skóre ≥ práh „Riziko" (výchozí 600) se počítá do červené dlaždice „PC v problémech". (Práh je laditelný a byl doladěn dle živého fleetu; nižší tier „Sledovat" v nastavení existuje, ale nezobrazuje se jako dlaždice.)
  • Kliknutím na dlaždici → se pod ní rozbalí tabulka s rozpadem (defaultně skrytá): skóre, počty critical/error/warning, počet typů chyb a počet dní s chybami. Dalším klikem na dlaždici nebo ✕ ji sbalíte.
  • Čísla v tabulce jsou proklikatelná do záložky Události: skóre otevře všechny chybové události daného PC za okno (a tooltip vysvětlí, jak se skóre počítá); crit / err / warn otevřou události toho PC profiltrované na danou závažnost. Jméno PC skočí do záložky Počítače.
  • Vše — okno, strop, prahy — se ladí v Nastavení → „PC v problémech".

What "Status" means now — reachability probe — the Status column used to merely reflect whether event-log collection succeeded: it ran only on monitored PCs, bucketed errors crudely, and eventually "froze", so supposedly-unreachable machines could still show Unknown or a green Active. A standalone reachability probe now runs: for every enabled, non-excluded PC (regardless of monitoring) it does a plain TCP connect to port 135 (RPC), falling back to 445 (SMB) — it is not an ICMP ping (Windows Firewall blocks ping by default in a domain). The probe runs on its own interval (reachability.interval_sec, default 300 s), independent of the main scan and its window, so Status stays fresh 24/7 including overnight and weekends. Toggle + interval live in Settings → "Network reachability (Status)".

How it reaches each PC

  1. Node-side TCP probe to ${name}:135 with 2s timeout. If fails → OFFLINE status, skip PS spawn entirely.
  2. Spawn powershell.exe -NoProfile -NonInteractive -Command "$ErrorActionPreference='Stop'; $OutputEncoding=[UTF8]; Get-WinEvent -ComputerName '${name}' -FilterHashtable @{ LogName='System','Application'; Level=1,2,3; StartTime=$startTime } | ConvertTo-Json"
  3. PS uses MS-EVEN6 RPC over SMB (enabled by default on domain-joined Windows; doesn't require WinRM listener)
  4. Failures classified by exception message:
    • RPC server is unavailable → rpc_unavailable (TCP/135 open but RPC layer blocked, typically firewall)
    • Access is denied → access_denied (RPC OK but Event Log Readers membership missing)
    • No events found → online with 0 events (counted as success)
    • otherwise → unknown

This section documents normal service behavior and monitoring blind spots for audit purposes (NIS2, ISO 27001 — operator/auditor will ask: "Do you have defined what is normal service behavior?").

What is monitored: every monitored PC (StartMode='Auto' AND State<>'Running'). Scan runs on the Periodic checks schedule (default every 90 min in window 06:00–18:00 Mon–Sun) plus the manual 🔧 Scan services button.

What is hidden by default (classified as legitimate state, not an alert):

  • Graceful exits (ExitCode = 0) — the ⚠ Only ExitCode != 0 filter is default ON. Hides rows where Win32 ExitCode = 0 (SCM cleanly stopped the service). This is the primary "stopped that shouldn't be" signal.
  • Trigger-start with graceful exit (registry TriggerInfo subkey + exit=0) — services designed as event-driven (login, network change, ETW), shut down cleanly after completing their task. The "Hide trigger-start" filter is default ON but only hides trigger-start with exit=0. A trigger-start that crashed (exit != 0) ALWAYS surfaces — that is a real failure.
  • Per-user services (service name matching _[a-f0-9]{4,12}$ LUID suffix) — bound to a user session, legitimately stopped when no user is logged on. Examples: CDPUserSvc, cbdhsvc, OneSyncSvc.
  • Compliant (matches a service_policy rule and complies with it) — default OFF, toggleable via "Hide compliant".

ExitCode semantics:

  • 0 = SCM-level graceful exit. For trigger-start typically means "completed the task". For classic Auto-stopped can mean "SCM timeout / startup failure where the service code returned 0 anyway" — so exit code alone is not an absolute gate.
  • != 0 = SCM recorded a failure. If ServiceSpecificExitCode is also != 0 (usually paired with Win32ExitCode=1066/ERROR_SERVICE_SPECIFIC_ERROR), the service itself reports an internal error code.
  • The ⚠ Only ExitCode != 0 filter isolates real crashes — useful for finding trigger-start services that actually failed.

Known blind spots (transparent disclosure):

  • No crash-loop detection — if a service starts and crashes rapidly (e.g. every 30 s with exit 0), it is not caught. Tracking state-flip frequency is Sprint 2 roadmap.

Server SKU gotcha

The Diagnostics-Performance channel is on by default on Windows 10/11 client, but off by default on Windows Server. Get-WinEvent on a disabled channel returns "There is not an event log on the X computer that matches". The collector matches this error pattern, classifies it as channel-disabled (separate counter, not a failure), and skips silently — no per-PC noise in the activity log, one aggregate line at end of run (e.g. "Perf scan done: 142 OK / 3 fail / 24 channel-disabled / +18 events").

To enable across the server fleet, push a GPO computer-startup script that runs:

wevtutil sl Microsoft-Windows-Diagnostics-Performance/Operational /e:true

Same pattern as the Services GPO script export. The channel starts recording on the next boot/shutdown/standby/resume cycle.

Why not continuous CPU history?

Windows does not natively retain a continuous CPU usage curve. Performance Monitor Data Collector Sets are off by default; SRUM stores per-process daily aggregates but isn't WMI-accessible. The Diagnostics-Performance channel was chosen because it is on by default, gives named-culprit attribution per incident, and reuses the existing RPC path with zero agent install. The tradeoff: you see Windows' opinion of which events were "slow" — the threshold is not configurable — and you don't see sustained workday usage, only diagnosed degradations.

This is the observer-not-executor pattern applied to telemetry: ITDashboard never installs an agent, never enables polling on the target, and never holds the target's truth — it only reads what Windows itself decided to record, on its own schedule.
