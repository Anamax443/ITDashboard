ITDashboard — DokumentaceITDashboard — Documentation

Interní IT provozní dashboard pro doménu AXINETWORK. Poskytuje napříč flotilou viditelnost eventlogu, AD-synchronizovaný inventář počítačů, monitorování diskového prostoru a per-PC klasifikaci dostupnosti pro ~225 doménových strojů. Internal IT operations dashboard for the AXINETWORK domain. Provides cross-fleet eventlog visibility, AD-synced computer inventory, disk space monitoring, and per-PC reachability classification for ~225 domain machines.

PřehledOverview

ITDashboard centralizuje čtyři pracovní postupy IT operátora v jednom dashboardu (tmavý/světlý motiv):

  • Eventlog napříč flotilou — stahuje události úrovně Warning/Error/Critical z každého monitorovaného PC do centrální DB; dashboard nabízí filtrování, vyhledávání, řazení, časové grafy a žebříček "noisy" PC.
  • AD-synchronizovaný inventář — drží aktuální seznam doménových počítačů s OS, posledním přihlášením, OU cestou a operátorem řízeným monitor toggle (perzistuje napříč syncs).
  • Monitorování disků — periodický DCOM skenování všech monitorovaných PC; konfigurovatelné thresholdy (% volného nebo GB volného nebo obojí); barevné progress bary per disk.
  • Klasifikace dostupnosti per PC — každý běh collectoru kategorizuje každý PC jako online / offline / RPC fail / access denied, prezentováno jako klikatelné dashboard karty pro drill-down.

ITDashboard centralises four IT-operator workflows in one dark-themed dashboard:

  • Cross-fleet eventlog visibility — pulls Warning/Error/Critical events from every monitored PC into a central DB; dashboard provides filtering, search, sorting, time-bucket charts, and noisy-PC ranking.
  • AD-synced inventory — keeps an up-to-date list of domain computers with OS, last logon, OU path, and operator-controlled monitor toggle (persists across syncs).
  • Disk space monitoring — periodic DCOM-based scan of all monitored PCs; configurable thresholds (% free or GB free or either); colored progress bars per drive.
  • Per-PC reachability classification — every collector run categorises each PC as online / offline / RPC fail / access denied, surfaced as clickable dashboard cards that drill down into filtered lists.

Jediná Node.js API služba na interním admin serveru + per-PC Electron klient (nebo browser přes Vite dev) pro IT specialisty. Single Node.js API service on the internal admin server + per-PC Electron client (or browser via Vite dev) for IT specialists.

KomponentyComponents

Server (apps/server)

  • Fastify + TypeScript HTTP API on port 4000
  • MSSQL via msnodesqlv8 driver (Windows-native, SSPI Integrated Auth; the pure-JS tedious driver fails with "untrusted domain" in domain environments)
  • Background tasks: unified periodic checks scheduler (eventlog / disk / services / perf), retention purge
  • PowerShell child processes for AD queries (Get-ADComputer), eventlog reads (Get-WinEvent -ComputerName), disk reads (Get-CimInstance via DCOM session)
  • Node-side TCP probe to port 135 before each PS spawn — fail-fast for offline targets, clean classification
  • In-memory ring buffer (500 entries) for activity log, polled by dashboard

Desktop client (apps/desktop)

  • Electron + React + Vite + TypeScript
  • Pure consumer of the API — no business logic, no direct AD/DB access
  • Auto-refresh dashboard every 30s; collector status polled every 2s during in-flight runs; activity log polled every 2s
  • 7 navigation tabs: Dashboard, Events, Computers, Services, Perf, Activity, Settings (+ external Docs link)
Server (apps/server): Fastify + TypeScript HTTP API na portu 4000; MSSQL přes msnodesqlv8 driver (Windows-native, SSPI Integrated Auth); background tasks (periodic checks scheduler, retention purge); PowerShell child procesy pro AD/eventlog/disk reads; Node-side TCP probe na 135 (fail-fast pro offline); in-memory ring buffer 500 entries pro activity log.

Desktop klient (apps/desktop): Electron + React + Vite + TypeScript; čistý consumer API; auto-refresh 30s; 7 navigačních tabů.

This section documents normal service behavior and monitoring blind spots for audit purposes (NIS2, ISO 27001 — operator/auditor will ask: "Do you have defined what is normal service behavior?").

What is monitored: every monitored PC (StartMode='Auto' AND State<>'Running'). Scan runs on the Periodic checks schedule (default every 90 min in window 06:00–18:00 Mon–Sun) plus the manual 🔧 Scan services button.

What is hidden by default (classified as legitimate state, not an alert):

  • No crash-loop detection — if a service starts and crashes rapidly (e.g. every 30 s with exit 0), it is not caught. Tracking state-flip frequency is Sprint 2 roadmap.

  • Whitelist is per-install, not centralized (GPO/CMDB sync) — OK for the solo-operator workflow, multi-admin operations is roadmap.

Server SKU gotcha

The Diagnostics-Performance channel is on by default on Windows 10/11 client, but off by default on Windows Server. Get-WinEvent on a disabled channel returns "There is not an event log on the X computer that matches". The collector matches this error pattern, classifies it as channel-disabled (separate counter, not a failure), and skips silently — no per-PC noise in the activity log, one aggregate line at end of run (e.g. "Perf scan done: 142 OK / 3 fail / 24 channel-disabled / +18 events").

To enable across the server fleet, push a GPO computer-startup script that runs:

wevtutil sl Microsoft-Windows-Diagnostics-Performance/Operational /e:true

Same pattern as the Services GPO script export. The channel starts recording on the next boot/shutdown/standby/resume cycle.

Why not continuous CPU history?

Windows does not natively retain a continuous CPU usage curve. Performance Monitor Data Collector Sets are off by default; SRUM stores per-process daily aggregates but isn't WMI-accessible. The Diagnostics-Performance channel was chosen because it is on by default, gives named-culprit attribution per incident, and reuses the existing RPC path with zero agent install. The tradeoff: you see Windows' opinion of which events were "slow" — the threshold is not configurable — and you don't see sustained workday usage, only diagnosed degradations.

This is the observer-not-executor pattern applied to telemetry: ITDashboard never installs an agent, never enables polling on the target, and never holds the target's truth — it only reads what Windows itself decided to record, on its own schedule.
