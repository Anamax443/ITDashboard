ITDashboard — Documentation

Internal AD & eventlog dashboard for IT operators. Eventlog analytics (Warning/Error/Critical focus), AD computer inventory, ad-hoc PowerShell/Python/C# script execution.

Overview

ITDashboard is a domain-aware diagnostic dashboard for the AXINETWORK environment. It centralises three IT-operator workflows:

  • Cross-fleet eventlog visibility — pulls Warning/Error/Critical events from all domain PCs into a central DB, exposes them via a unified dashboard with filtering, sorting, and trend charts.
  • AD inventory — keeps a synchronised list of domain computers with OS version, last logon, and enabled state.
  • Audit-friendly script execution — versioned catalog of PS/Python/C# scripts runnable on demand with parameters, output captured to DB.

It runs as a single Node.js API service on the internal admin server, with a per-PC Electron client distributed to IT specialists.

Deployment workflow

Push-to-deploy via GitHub Actions self-hosted runner. No manual server interventions.

local edit (d:/git/ITDashboard) ↓ git push origin main GitHub Actions ↓ self-hosted runner on 10.8.2.213 picks up job robocopy → C:\Apps\ITDashboard (runtime location) ↓ npm install + typecheck + build + migrate ↓ net stop ITDashboardAPI && net start ITDashboardAPI
Commit hash in topbar: The deployed commit SHA is displayed in the dashboard topbar. If after a push the hash doesn't change, the deploy failed silently. Click the hash to open the GitHub commit page.

Components

Server (apps/server)

  • Fastify + TypeScript HTTP API on port 4000
  • MSSQL via msnodesqlv8 driver (Windows-native, supports SSPI Integrated Auth; the pure-JS tedious driver fails with "untrusted domain" in domain environments)
  • Background tasks: eventlog collector loop, retention purge
  • PowerShell child processes for AD queries and remote event reading

Desktop client (apps/desktop)

  • Electron + React + Vite + TypeScript
  • Pure consumer of the API — no business logic, no direct AD/DB access
  • Auto-refresh every 30s; collector status polled every 2s during in-flight runs

How it reaches each PC

The collector uses Get-WinEvent -ComputerName <PC> which talks MS-EVEN6 RPC over SMB. This protocol is enabled by default on domain-joined Windows machines and is more reliable than WinRM/PSRemoting in environments where WinRM hasn't been explicitly configured by GPO.

Required permission on target PC: svc-itdashboard must be a member of the local Event Log Readers group on each target PC. The standard way to deploy this is via Group Policy: 
Computer Configuration → Preferences → Control Panel Settings →
  Local Users and Groups → Group "Event Log Readers" (Built-in) →
  Add member: AXINETWORK\svc-itdashboard
After GPO update (gpupdate /force), collector will start succeeding.

AD sync

Calls Get-ADComputer -Filter * against the domain controller and MERGE-upserts into the computers table.

  • Computers no longer present in AD are soft-disabled (enabled = 0) instead of deleted — preserves historical events.
  • History of every sync persists in ad_sync_runs table (fetched/inserted/updated/removed counts + duration).
  • Triggered manually from Computers tab → ↻ Sync from AD. Currently not auto-scheduled (planned).

Script runner

Versioned catalog at scripts/ in the repo. Each entry described in scripts/manifest.json with name, language, path, and parameters.

Currently shipped:

  • ad-user-info — Get-ADUser + group membership
  • pc-services — Remote PC services list

Run history (invoker, target, params, stdout/stderr, exit code) persists in script_runs table.

Troubleshooting

Collector failing on all PCs with "Connecting to remote server failed"

Cause: Old code used WinRM/PSRemoting which is rarely enabled by default. Fix: Already deployed switch to Get-WinEvent -ComputerName (RPC). If still failing fleet-wide after fix, svc-itdashboard needs Event Log Readers membership on target PCs — deploy via GPO (see Collector section).

Deploy succeeds but new endpoint returns 404

Cause: Robocopy in deploy.yml failed silently (typically NTFS access denied — svc-itdashboard lacks Modify rights on C:\Apps\ITDashboard). Fix:

icacls 'C:\Apps\ITDashboard' /grant 'AXINETWORK\svc-itdashboard:(OI)(CI)M' /T

SQL connection fails with "untrusted domain"

Cause: Using pure-JS tedious driver instead of msnodesqlv8. Fix: Already in code — pool.ts imports mssql/msnodesqlv8.js.

GitHub Actions workflow fails immediately with UnauthorizedAccess on .ps1

Cause: Group Policy enforces AllSigned ExecutionPolicy on AXINETWORK servers. Runner saves each step as temp .ps1, which can't load. Fix: Already in deploy.yml — defaults.run.shell: cmd.

SQL_INSTANCE GitHub variable can't be empty

Cause: The dedicated SQL server uses the default instance (no \BCNEW), but GitHub Actions variables can't be empty. Fix: Set SQL_INSTANCE = _ (sentinel value); server interprets _ or empty as "no instance".

PCs disappear from list after Sync from AD

Behaviour, not bug: PCs no longer present in AD are soft-disabled (enabled = 0), shown with reduced opacity at the bottom of the Computers tab. This preserves their historical events. Use the Status filter to focus on active PCs only.
