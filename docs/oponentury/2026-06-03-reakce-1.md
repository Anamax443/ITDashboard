## 1. Architektura a principy

### 1.1 Observer-not-executor — AKCEPTUJI varování

Souhlasím. Tlak na "stiskni a oprav" reálně přijde. Reviewer's návrh "samostatný nástroj s vlastním threat modelem" pro remediaci je správný — kdyby jsme chtěli masově restartovat službu na 80 PC, není to ITDashboard, je to **ITDashboard-Executor** nebo prostě GPO/Ansible.

**Akce:** Doplnit do ARCHITECTURE.md tvrdou policy:
> Žádné tlačítko v ITDashboardu, které mění stav vzdáleného PC. Pokud
> nutné, samostatný nástroj s vlastní auth a per-akce auditem.

### 1.2 Polling vs push — DÍLČÍ akceptace

Souhlasím s "push agent odmítám z principu". Disk-critical fast loop ale nepřebírám hned:

- Disk-critical na 5 minut by znamenal per-PC speciální scheduling. To otevírá komplikaci "pro tento PC kratší interval", což je první step k per-PC override mechanizmu. Pro 1-2 critical PC v ráně OK, pro fleet to chce design.
- Real win pro disk-critical: alert (Resend / Teams webhook) když threshold překročen, ne vyšší polling. Operator chce signál, ne víc samplů.

**Akce:** Backlog P1 — alerting pro disk-critical (existující P1 item). Speciální per-PC interval = ne.

### 1.3 Single point of failure + healthcheck — AKCEPTUJI

`/health` má vracet DB stav (viz bod 7.1 níže). Centron probe = dohoda s IT.

**Akce:** P0 — rozšířit `/health` (viz 7.1).

### 1.4 Per-feature plugin vs flat routes — souhlas s reviewerem

Reviewer doporučuje **nedělat** plugin refactor. Souhlasím. Zůstává flat.

## 2. Bezpečnost

### 2.1 API bez autentizace — AKCEPTUJI substantia, DÍLČÍ k řešení

Reviewer má pravdu, že "doména jako trust boundary v 2026" je slabé.
Ale "30 minut práce" je podstřel.

**Co reviewer navrhuje** (pre-shared token v `process.env.API_TOKEN`):
- Token v env var → kdo ho viděl, má všechno
- Žádná rotace, žádná expirace
- Token distribuce na operátorský browser = zapsat ho do localStorage manuálně nebo přes URL param
- Pokud uniká přes log / network capture, není revokace

Pro low-effort "lepší než nic" to OK je. Ale není to skutečný auth.

**Co bych udělal místo toho:**
- Token-per-operator (každý IT operátor má vlastní), uložený v DB s `created_at`, `expires_at`, `last_used`, `revoked`.
- Frontend si token uloží do `localStorage` po prvním zadání (UI flow).
- Server expiraci enforce, rotace volitelně kdykoli.
- Loopback (`127.0.0.1`) bypass zůstává pro self-calls.

Náročnost: ~4-6 hodin, ne 30 min. Ale je to reálná auth boundary, ne UX gate s headerem.

**Akce:** P0 — implementovat per-operator token auth. Pre-shared token v env odmítám jako náhradu, jen jako "lepší než žádný" interim.

### 2.2 Firewall Domain profile vypnutý — AKCEPTUJI detection

Souhlas s "warning banner v UI". Není to v naší moci to zapnout (GPO), ale operátor by měl vědět že OS-level gate je inertní.

**Akce:** P1 — boot-time check + persistent warning v topbar/Settings.

### 2.3 Audit log uživatelských akcí — DÍLČÍ akceptace

Souhlasím že chybí. Ale **rozsah** je otázka:

- "Kdo si zobrazil Computers tab" = každý GET request → spousta zápisů, většina šum
- "Kdo stáhl GPO script" / "Kdo spustil Run all" / "Kdo upravil Settings" = state-changing actions → real signal

**Akce:** P2 — audit log **jen pro state-changing actions** (PUT/POST), ne pro každý GET. `user_activity` tabulka s ip, action_type, target, timestamp. Read-only auditing až když tým > 1.

### 2.4 DPAPI credentials vault — AKCEPTUJI recovery doc

Souhlas. Recovery flow neexistuje. Dnes není kritické (vault je prázdný, `credentials` tabulka má 0 řádků), ale měl by být dokumentován pre-emptive.

**Akce:** Doplnit do SETUP-SERVER.md recovery procedure (export blob, fallback flow).

### 2.5 Loopback always allowed — souhlas s "dokumentovat"

To je v HANDOFF.md / ARCHITECTURE.md zmíněné. Reviewer's "akceptovatelné, ale dokumentovat" = OK.

## 3. Databáze a migrace

### 3.1 Retention scheduler — AKCEPTUJI okamžitě

Toto je největší zaslepené místo. `sp_purge_old_*` procedures existují ale nikdo je nevolá. Data rostou neomezeně. **Den 1 priorita.**

**Akce:** P0 — `setInterval` daily v `checks-runner.ts` nebo separate `retention-runner.ts`. Volá `sp_purge_old_events 90` + `sp_purge_old_activity 30` v 02:00. ~1h práce.

### 3.2 GO separator / EXEC pattern — DÍLČÍ akceptace

Souhlasím že `EXEC('CREATE PROCEDURE ...')` je křehký. Ale reviewer's návrh
"rozdělit na příkazy podle `;` (pozor na string literály)" je horší —
psát SQL parser je open invitation pro chyby.

**Lepší alternativa:** Jeden `.sql` file = jeden batch. Pro stored procedures
samostatný `.sql` file (`020a_activity_log_table.sql`, `020b_activity_log_purge_proc.sql`).
Migration runner pak nemusí umět split.

**Akce:** P2 — refaktor migration konvence (1 file = 1 batch).

### 3.3 Clustered index na events — souhlas

Reviewer's "monitorovat fragmentaci" je správné. Není to dnes problém (2M řádků), za rok asi ano.

**Akce:** P2 — týdenní MSSQL maintenance plan pro index rebuild.

### 3.4 Reserved word incident — AKCEPTUJI hlavní bod

> "Že to prošlo do produkce je selhání lokálního testování."

Souhlas. Bez lokálního MSSQL nelze migrace pre-flightnout.

**Akce:** P1 — instalovat MSSQL Developer Edition lokálně (free, 10GB limit OK), `npm run migrate:local` script.

## 4. Deployment a CI/CD

### 4.1 Topbar SHA lže — AKCEPTUJI okamžitě

Reviewer popsal přesně. **To je top diagnostic priorita.**

Reviewer's návrh (`build:version` script generuje `version.ts` s GIT_SHA constant) je správný a jednoduchý. Beru s drobnou úpravou:

```ts
// apps/server/scripts/build-version.mjs
import { execSync } from 'child_process';
import { writeFileSync } from 'fs';
const sha = execSync('git rev-parse HEAD').toString().trim();
const built = new Date().toISOString();
writeFileSync('apps/server/src/build-info.ts',
  `export const BUILD_SHA = '${sha}';\nexport const BUILT_AT = '${built}';\n`);
```

Generated soubor v gitignore. Volá se v `npm run build` před `tsc`.

**Akce:** P0 — implementovat dnes.

### 4.2 Smoke test po deployi — AKCEPTUJI

Reviewer's `curl /version | findstr "%GITHUB_SHA%"` je správný idiom.
Drobná korekce: porovnání s `$GITHUB_SHA` short SHA (7 znaků), ne full.

**Akce:** P0 — přidat do deploy.yml jako final step po `sc start`.

### 4.3 Self-hosted runner SPOF — PROPORČNÍ POZNÁMKA

Reviewer's návrh fallback "ručního deploy script" je rozumný, ale priorita nízká.
Když runner spadne, deploy jen čeká — neztrácíme produkční trafik.

**Akce:** Backlog P2 — `scripts/manual-deploy.ps1` (download zip, robocopy, sc restart). 1h, low pri.

### 4.4 `shell: cmd` jako obejití GPO — ODMÍTÁM mischaracterization

Reviewer to nazval "obejití firemní GPO policy" a "red flag z audit trail". To je nesprávně popsané:

- GPO `AllSigned` ExecutionPolicy se vztahuje **pouze na PowerShell**. Nezakazuje `cmd`.
- `cmd` je legitimní shell, GPO ho nezakazuje.
- Použití `cmd` ≠ obcházení AllSigned. Je to **použití jiného nástroje, který té politice nepodléhá**.

Code-signing PS scriptů je teoreticky čistší, ale:
- Cert distribuce per dev workflow je overhead
- Re-signovat při každé změně automatovaného PS step v workflow je nepraktické
- Reálně to neřeší žádné security riziko (AllSigned je proti malicious PS, ne proti DevOps automation)

**Akce:** Žádná. Komentář v deploy.yml ať je explicitní:
> # Using cmd shell because AXIMA GPO enforces AllSigned for PS. This is
> # not bypassing policy — cmd is a separate, non-PS shell. PS-specific
> # GPO does not apply.

### 5.5 CSP `unsafe-inline` na /docs — ODMÍTÁM proporčně

Reviewer označil za "XSS riziko". Proporčně:

- `/docs` je **static HTML soubor** servírovaný z disku, žádný user input nikdy nedosáhne na content
- Žádný cookie / session / privileged operace na `/docs`
- Inline JS v dashboard.html je 1 řádek `window.print()`

Reviewer's návrh "refactor na external script + nonce" je technicky správný, ale:
- Nonce vyžaduje per-request CSP header s freshly generated nonce
- Pro static HTML file servírovaný z disku to znamená šablonu + render middleware
- Náročnost 2-3h pro nulový reálný security benefit (žádný XSS attack vector existuje)

**Akce:** Žádná. Pokud se v budoucnu `/docs` stane dynamickou (např. operator-editable notes), pak nonce. Dnes přepych.

## 6. Backend a collectory

### 6.1 PS skripty extrakce do `.ps1` — AKCEPTUJI

Reviewer's návrh je správný. `\d` bug to ukázal. Bonus signed scripts (pokud bychom code-signing zavedli).

Akce:
1. Vytvořit `apps/server/scripts/` adresář s `.ps1` files
2. Loaderem v Node načíst + passthrough args
3. Per collector jeden file

Náročnost: reviewer odhadl 4h. Realisticky 6-8h (testování per-collector).

**Akce:** P1.

### 6.2 Per-event INSERT vs table-valued parameter — DÍLČÍ akceptace

Reviewer's "při 500 PC to bude bolet" je odhad. Skutečnost: dnes ~3000 INSERTů
za cyklus per ~60s = 50 INSERTs/sec. MSSQL pohodlně 5000+/sec.

Při 500 PC = ~15000 INSERTů per ~5min = 50 INSERTs/sec stejně.
Spíš bottleneck je PS spawn (5 paralelně) než DB.

**Akce:** P3 — když budem hostit 500+ PC, ano. Dnes je to non-issue.

### 6.3 `runInFlight` reset — souhlas s reviewerem

Reviewer's analýza je správná. `finally` resetuje, OK. Žádná akce.

### 6.4 Fire-and-forget activity log — DÍLČÍ akceptace

Reviewer's "pokud DB spadne na 5 minut, ztratíte všechny logy" je správně.
Ale "memory queue + retry + dump do souboru po N pokusech" znamená:

- Queue management v paměti
- Disk write fallback (kde? `C:\Apps\ITDashboard\fallback.log`?)
- Recovery logiku při restartu (read fallback + flush do DB)

Pro produkci s SLA ano. Pro náš případ (DB spadne nečekaně, restart 5 min): co se v té době neuložilo, je v in-memory ring buffer (500 entries) přístupný přes `/activity/log`. Operátor to vidí, jen ne historicky.

**Akce:** P2 — log do `C:\Apps\ITDashboard\fallback.log` jako simple file appender, pokud DB write hodí. Bez queue ani recovery flow. To je 80/20.

### 6.5 Backoff pro failed PCs — DÍLČÍ akceptace

Reviewer's "po 5 failures za 1h, po 10 za 6h, po N za 24h" je rozumný.
Ale dnešní pattern "10 failures → trvale skip → operator reset" má výhodu:
operator **vidí, že něco je rozbité** a aktivně to opraví. Exponenciální
backoff to "vyřeší" automaticky a operátor neví že PC byl 24h offline.

**Lepší řešení:** Backoff ano, ale s visible signalizací: PC v UI ukazuje
"backed off until 14:00" v Status sloupci. Operator vidí, kdy zkusíme znovu.

**Akce:** P1 — backoff s visible state v UI. Reviewer's exponential pattern přebírám, ale s `next_retry_at` columnem v `computers`.

### 6.6 Server SKU enable channel button — AKCEPTUJI

Reviewer's návrh "export GPO script pro `wevtutil sl .../e:true`" je správný,
následuje pattern Services GPO export.

**Akce:** P2 — nový button v Perf tabu "📤 Enable channel on servers (GPO script)".

  je tam.

## 12. Co reviewer nezmínil ale měl by

Self-review co reviewer přehlédl nebo zmínil málo:

1. **Žádný error budget / SLO definovaný.** Co je "OK" downtime za měsíc? Není.
2. **Žádná dokumentace cílového user persona.** ARCHITECTURE.md říká "IT operátor" ale neříká úroveň zkušenosti, jaké nástroje paralelně používá, co tady spotřebovává.
3. **Žádný changelog public-facing.** HANDOFF.md je interní timeline. Pokud přijde druhý operátor, neuvidí "co se změnilo za týden" snadně. `CHANGELOG.md` per release tag.
4. **Žádný versioning policy pro `/api`.** Když změním shape `/computers` response, breaking change pro frontend bundle s jiným cache TTL. Dnes ne-issue (frontend a server deploy spolu), ale pokud bychom oddělili (např. desktop Electron klient s vlastním upgrade cyklem), problém.
